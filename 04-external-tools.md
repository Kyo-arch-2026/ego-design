# 04. External Tools — 外部ツール選定の設計

E.G.Oは「車輪の再発明をしない」を原則とする。外部ツールを選ぶ時も「なぜそれか」「なぜ他はダメか」を明示する。

## 1. 入力基盤：Hermes Agent

採用する。E.G.Oの入力・インターフェース層として使う。

Hermes Agentは、Nous Researchが開発したオープンソースのAIエージェント基盤。セッションをまたいだ記憶と、プロバイダー非依存を設計の核に持つ。

なぜHermesを選んだか：
- セッションをまたぐ記憶 → 原則1（意志の文脈を劣化させない）と一致
- プロバイダー非依存 → 原則2（LLMは補助役）と一致

なぜ他のツールではないか：

| ツール | 不採用理由 |
|---|---|
| OpenClaw | Hermesへの公式移行パスがあり、Hermesが後継 |
| 自前実装 | 入力層より正本管理に集中すべき（原則3） |
| LangChain | フレームワーク依存が強くロックインのリスク |

## 2. データベース：PostgreSQL + pgvector

採用する。

採用理由：
- 正本管理に必要な書き込みの信頼性（ACID）が高い
- pgvectorでベクトル検索もできる
- 1つのDBで全文検索・ベクトル検索・正本管理が完結

なぜ他ではないか：

| 選択肢 | 不採用理由 |
|---|---|
| SQLite | 拡張で検索は可能だが、本番実績と信頼性で劣る |
| MongoDB | 正本管理に必要なトランザクションの堅牢性が弱い |
| 専用ベクトルDB（DynamoDB等） | キー検索が前提で複雑な状態管理に不向き。DB種別を増やしたくない（原則3） |

## 3. LLM：抽象化レイヤーで切り替え可能に

特定プロバイダーに依存しない。

| 用途 | プロバイダー |
|---|---|
| 深い思考 | Claude Pro |
| 前処理・分類 | cheap LLM |
| ローカル実行 | Ollama |

## 4. なぜ個人開発でAWSを使わないか

E.G.Oは個人の判断データを扱うため、データを手元に置く設計を選んでいる。これは「常駐コンポーネントを増やさない」（原則3）という設計原則とも一致する。

ただし、将来複数デバイスでの利用が必要になった場合は、RDS（PostgreSQL）+ EC2/Lambdaへの移行パスを想定している。今は不要な常駐コストを避けるという、コスト合理性に基づいた判断である。
