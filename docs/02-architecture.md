# 02. Architecture — システム設計

## 全体構成

E.G.Oは「拡張型モジュラーモノリス + プラグインアーキテクチャ」を採用する。1つのアプリケーションとしてシンプルに動作しながら、内部は機能ごとにモジュール化され、後から機能を追加しやすい構造を持つ。

## レイヤー構成

E.G.Oは4つの層で構成される。

### 入力層 / インターフェース

- Discord、Telegram、CLIなどから入力を受け取る
- Hermes Agentが入力とスキル実行を担う
- ユーザーとの接点となる層

### E.G.O Core（中枢）

意志の保護・正本管理・文脈の蓄積を担う中核。以下のモジュールで構成される。

- **Thought Structurer**：自由記述を要約・課題・選択肢・次のアクションに構造化する（原則1・2）
- **Session Manager**：文脈を前面化し、どの情報を参照するかを制御する
- **Approval Flow**：AIの解釈を人間が承認する仕組み（candidate → active）
- **Source of Truth**：正本を管理し、上書きせず履歴として残す
- **Audit Log**：全操作を記録する

### データ層（4層構造）

| 層 | 役割 |
|---|---|
| Canonical Layer（正本層） | 最終的に信じる情報 |
| Memory Layer（記憶層） | 中長期の記憶 |
| Observation Layer（観測層） | 外部ソースからの観測 |
| Raw Evidence Layer（原文層） | 復元・監査のための元データ |

ストレージの役割分担：

| ストレージ | 役割 |
|---|---|
| PostgreSQL + pgvector | 正本・記憶・メタデータ・全文/ベクトル検索 |
| Valkey | セッションキャッシュ・短期状態 |
| Object Storage | 原文（raw HTML・PDF・会話原文） |

### LLM層（抽象化レイヤー）

特定プロバイダーに依存しない。

| 用途 | プロバイダー | 理由 |
|---|---|---|
| 深い思考・handoff | Claude Pro | 長文・思想的議論に強い |
| 前処理・分類（並列） | cheap LLM | コスト・速度のバランス |
| ローカル実行 | Ollama経由 | プライバシー・オフライン対応 |
