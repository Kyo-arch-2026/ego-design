# 02. Architecture — システム設計

## 全体構成

E.G.Oは「拡張型モジュラーモノリス + プラグイン + ヘキサゴナル（ポート＆アダプタ）」の考え方を採用する。1つのアプリケーションとしてシンプルに動作しながら、内部は機能ごとにモジュール化され、後から機能を追加しやすい構造を持つ。

さらに、外部と接する3つの層（入力・LLM・ストレージ）を抽象ポートで隔離し、具体的な技術（Hermes／Claude／SQLite など）はその背後のアダプタに閉じ込める。E.G.O Core はアダプタの背後にある実体を知らない。これにより、特定技術の更新に全体が振り回されず（保守の手間削減）、障害をアダプタ単位で切り分けられ（障害点の切り分け容易化）、新しい入力元・プロバイダー・ストレージをアダプタ追加だけで足せる（機能追加容易化）。これは「機能性こそが能動性の前提」という思想と、原則3（常駐コンポーネントを増やさない）の帰結である。

## レイヤー構成

E.G.Oは4つの層で構成される。外部と接する層は、抽象ポートを介して Core と接続する。

### 入力層 / インターフェース

- CLI、Discord、Telegram などから入力を受け取る
- すべての入力元は **Input Port（抽象境界）に適合する「入力アダプタ」** として実装する
- Core は Input Port だけを知り、どの入力元かを知らない
- Hermes Agent は「入力アダプタの一選択肢」であり、本体には依存しない（詳細は `04-external-tools.md`）
- ユーザーとの接点となる層

### E.G.O Core（中枢）

意志の保護・正本管理・文脈の蓄積を担う中核。外部技術を知らない。以下のモジュールで構成される。

- **Thought Structurer**：自由記述を要約・課題・選択肢・次のアクションに構造化する（原則1・2）。LLM 呼び出しは LLM Port 経由で行う
- **Session Manager**：文脈を前面化し、どの情報を参照するかを制御する。参照は Store Port 経由
- **Approval Flow**：AIの解釈を人間が承認する仕組み（candidate → active）
- **Source of Truth**：正本を管理し、上書きせず履歴として残す。永続化は Store Port 経由
- **Audit Log**：全操作を記録する

### 抽象化・アダプタ基盤

外部と接する3つの層を抽象ポートで隔離する。Core はポートだけを知り、その背後のアダプタ（具体技術）を知らない。

| ポート | 役割 | 適合するアダプタ |
| --- | --- | --- |
| Input Port | 入力を受け取る抽象境界 | CLI（1.0）／Discord・Telegram・Hermes（1.5） |
| LLM Port | LLM 呼び出しの抽象境界 | Claude（1.0）／cheap LLM・Ollama（1.5） |
| Store Port | 永続化・検索の抽象境界 | SQLite（1.0）／PostgreSQL+pgvector（1.5） |

依存の向きは常に「アダプタ → ポート → Core」。Core からアダプタへの依存は持たない。これにより、アダプタ（外部技術）が変わっても Core は無改修で済む。

### データ層（4層構造）

Store Port の背後にあるアダプタが保持する。Core はテーブルの物理形式を知らず、ドメイン概念で扱う。

| 層 | 役割 |
| --- | --- |
| Canonical Layer（正本層） | 最終的に信じる情報 |
| Memory Layer（記憶層） | 中長期の記憶 |
| Observation Layer（観測層） | 外部ソースからの観測 |
| Raw Evidence Layer（原文層） | 復元・監査のための元データ |

ストレージの役割分担：

| ストレージ | 役割 | フェーズ |
| --- | --- | --- |
| SQLite | Store Port の Phase 1.0 アダプタ実装。正本・履歴・監査ログの保持 | 1.0 |
| PostgreSQL + pgvector | Store Port のアダプタ追加。正本・記憶・メタデータ・全文/ベクトル検索 | 1.5〜 |
| Valkey | セッションキャッシュ・短期状態 | 1.5〜 |
| Object Storage | 原文（raw HTML・PDF・会話原文） | 1.5〜 |

### LLM層（抽象化レイヤー）

LLM Port を介して呼び出す。特定プロバイダーに依存しない。

| 用途 | プロバイダー | 理由 |
| --- | --- | --- |
| 深い思考・handoff | Claude | 長文・思想的議論に強い |
| 前処理・分類（並列） | cheap LLM | コスト・速度のバランス |
| ローカル実行 | Ollama経由 | プライバシー・オフライン対応 |

## データの状態遷移

正本は直接上書きせず、状態を遷移させる。

```
candidate（候補） → active（有効） → superseded（置換済）

↓

invalid / archived
```

なぜ状態遷移か：

- 「なぜ考えが変わったか」を残せる
- 復元できる状態を保てる
- 誤情報の混入を防げる

これは原則1（意志の文脈を劣化させない）の具体的な実装である。既存のLLMは過去の情報をフラットに扱い、古くなった判断もそのまま参照してしまう。E.G.Oは情報に時間軸を持たせ、常に「今有効な正本」だけを参照できるようにする。詳細は `03-data-design.md` を参照。

## 設計を貫く4原則との対応

| モジュール | 対応する原則 |
| --- | --- |
| Thought Structurer | 原則1（原文保持）・原則2（承認前提） |
| Approval Flow | 原則2（LLMは補助役） |
| Source of Truth | 原則1（履歴を残す） |
| 抽象ポート（Input/LLM/Store） | 原則2（プロバイダー非依存）・原則3（技術非依存・疎結合） |
| Audit Log | 原則1（全操作の痕跡） |
| プラグイン／アダプタ構造 | 原則3（常駐コンポーネントを増やさない） |

## Phase 1.0での実装範囲

Phase 1.0では、3つの抽象ポート（Input/LLM/Store）を定義し、それぞれ1つのアダプタ（CLI／Claude／SQLite）だけを実装する。実装する範囲（設計確定・実装着手前）：

- 思考の構造化
- 承認フロー（candidate → active）
- 正本管理（Store Port 経由。Phase 1.0 アダプタは SQLite）
- 監査ログ
- 3つの抽象ポートと Phase 1.0 アダプタ（CLI／Claude／SQLite）

将来のPhaseで実装：

- PostgreSQL + pgvector アダプタの追加（Phase 1.5。Store Port へのアダプタ追加のみで移行）
- 並列cheap LLM処理（Phase 1.5）
- 入力アダプタの拡充（Discord/Telegram/Hermes、Phase 1.5）
- AIの解釈過程の可視化（Phase 1.5以降の課題）

Phase 1.0は「思想が機能するか」の検証を目的とする。なお本リポジトリは現在、設計フェーズにあり、コードによる実装はこれからである。公開しているのは設計ドキュメントである。
