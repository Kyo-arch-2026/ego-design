# E.G.O 機能分解書（Function Breakdown）

**作成日**：2026-07-18
**作成者**：石井恭平
**対象**：個人用AI基盤「E.G.O」
**目的**：E.G.O が持つ機能を層ごとに体系的に洗い出し、設計・実装の状態を明示する。転職活動の成果物として、また実装の作業計画として用いる。

## 0. 読み方

- **層（大分類）**：E.G.O の 4 レイヤー構成（入力層／Core／データ層／LLM 層）を基準に、機能を束ねている。分類の考え方は、機能を「入口・中核・記憶・拡張・運用」で層状に整理する Hermes Agent の構造を参考にした。
- **状態**の凡例：
  - 📐 **設計済**：設計ドキュメントで方針が確定している。コードは未着手。
  - 🔨 **実装予定（Phase 1.0）**：Phase 1.0 でコードを書く対象。
  - 📋 **計画中（Phase 1.5以降）**：将来フェーズに予定し、現時点で未着手。
- **現状の前提**：本書作成時点で、E.G.O は **設計フェーズ**にあり、コードや動作するものはまだ存在しない。GitHub 上の設計ドキュメントが唯一の成果物である。したがって本書の状態表記は「どこまで設計が固まり、どこから実装・構想か」を示すものであり、動作状況を示すものではない。
- **対応原則**：各機能が E.G.O の 4 原則のどれを根拠に存在するかを示す。
  - 原則1＝意志の文脈を劣化させない／原則2＝LLM（大規模言語モデル）は補助役に徹する／原則3＝常駐コンポーネントを増やさない／原則4＝使うほど自分への解像度が上がる

> 思想的背景は `docs/00-origin-story.md`・`docs/01-philosophy.md`、技術選定根拠は `docs/04-external-tools.md` を参照。本書は「何を作るか」の一覧に徹する。

## 2. 機能分解（詳細）

### A. 入力・インターフェース層

ユーザーとの接点。Hermes Agent（Nous Research 製のオープンソース AI エージェント基盤）を採用し、入力受付とスキル実行を担わせる。

| ID | 中分類 | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|---|
| A-1 | 入力受付 | CLI（コマンドライン）からの入力受付 | 🔨 実装予定 | 原則3 |
| A-2 | 入力受付 | Discord／Telegram からの入力受付 | 📋 計画中 | 原則3 |
| A-3 | 入力受付 | 音声メモの文字起こし入力 | 📋 計画中 | 原則3 |
| A-4 | セッション連続性 | プラットフォームをまたいだ会話の継続 | 📋 計画中 | 原則1 |
| A-5 | スキル実行 | Hermes のスキル機構経由での処理実行 | 📋 計画中 | 原則3 |
| A-6 | 入力正規化 | 各入力元の形式差を Core 向けに整える | 🔨 実装予定 | 原則1 |

> **設計メモ**：Phase 1.0 は CLI のみで検証する。多チャネル化（A-2〜A-5）は Hermes 統合を行う Phase 1.5 の範囲。入力層を自前実装せず Hermes に委ねるのは、正本管理に集中するため（原則3）。

---

### B. 思考構造化（Thought Structurer）

自由記述を「要約・課題・選択肢・次のアクション」に構造化する。ここで生成された内容は**まだ正本ではなく候補（candidate）**であり、人間の承認を経て初めて正本になる。

| ID | 中分類 | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|---|
| B-1 | 構造化 | 自由記述の要約生成 | 🔨 実装予定 | 原則2 |
| B-2 | 構造化 | 課題の抽出 | 🔨 実装予定 | 原則2 |
| B-3 | 構造化 | 選択肢の列挙 | 🔨 実装予定 | 原則2 |
| B-4 | 構造化 | 次のアクションの提示 | 🔨 実装予定 | 原則2 |
| B-5 | 原文保持 | 構造化前の元テキストを原文として保持 | 🔨 実装予定 | 原則1 |
| B-6 | 候補化 | 構造化結果を candidate 状態で登録 | 🔨 実装予定 | 原則1・2 |

> **設計メモ**：構造化は LLM が行うが、その出力を正本に自動昇格させない（原則2）。B-5 の原文保持により、後から「AI が何をどう変換したか」を復元できる。

---

### C. 正本管理・状態遷移（Source of Truth）★ E.G.O の核心

情報を上書きせず、状態を遷移させることで「今有効な正本」だけを参照可能にする。この会話で固めた **「正本テーブルと履歴テーブルの分離」** をここに反映する。

#### C-1. データ構造

| ID | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|
| C-1-1 | 正本テーブル（`canonical_facts`）：`active` のみを保持する小さいテーブル | 🔨 実装予定 | 原則1 |
| C-1-2 | 履歴テーブル（`fact_revisions`）：candidate／superseded／invalid／archived を保持する追記専用ログ | 🔨 実装予定 | 原則1 |
| C-1-3 | 事実カード型の主キー設計（1 判断＝1 カードを基本単位とする） | 🔨 実装予定 | 原則4 |
| C-1-4 | トピック束ねタグ（`topic_tag`）による後付けグルーピング用カラム | 📋 計画中 | 原則4 |
| C-1-5 | 時間軸カラム（`valid_from`／`valid_until`）の保持 | 🔨 実装予定 | 原則1 |
| C-1-6 | 自由タグ（`tags`）の保持 | 🔨 実装予定 | 原則4 |

#### C-2. 状態遷移

| ID | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|
| C-2-1 | candidate → active（承認による正本化） | 🔨 実装予定 | 原則2 |
| C-2-2 | active → superseded（新しい正本への置換。旧行は履歴テーブルへ退避） | 🔨 実装予定 | 原則1 |
| C-2-3 | → invalid（誤情報として無効化） | 📋 計画中 | 原則1 |
| C-2-4 | → archived（保管） | 📋 計画中 | 原則1 |
| C-2-5 | 状態遷移時に「なぜ変わったか」の理由を記録 | 📋 計画中 | 原則1・4 |
| C-2-6 | 過去状態への復元（履歴テーブルからの巻き戻し） | 📋 計画中 | 原則1 |

> **設計メモ**：正本テーブルを小さく保つことで、AI 参照時の曖昧さとハルシネーションを抑える。履歴は履歴テーブルに完全保存し、人間の追跡性を担保する。型（カード型かトピック型か）は使用感とレイテンシーを見て確定する方針のため、C-1-3 でカードを主キーに置き、C-1-4 のタグで後からトピック型ビューを構成できる形にしている（今は型を決め打ちしない）。

### D. 承認フロー（Approval Flow）

AI の解釈（candidate）を、人間が承認して初めて正本（active）にする。原則2「LLM は補助役に徹する」を最も直接に体現する機能。

| ID | 中分類 | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|---|
| D-1 | 承認 | candidate の内容を人間に提示 | 🔨 実装予定 | 原則2 |
| D-2 | 承認 | 承認操作による active 昇格 | 🔨 実装予定 | 原則2 |
| D-3 | 却下 | candidate の却下（正本化しない） | 🔨 実装予定 | 原則2 |
| D-4 | 修正 | 承認前に人間が内容を修正 | 📋 計画中 | 原則1・2 |
| D-5 | 承認記録 | 誰がいつ承認／却下したかを監査ログへ記録 | 🔨 実装予定 | 原則1 |

> **設計メモ**：D-4（承認前修正）は「意志を劣化させない」ために重要だが、Phase 1.0 では承認／却下の二択を先に固め、修正は Phase 1.5 に回す。

### E. 検索・参照（Session Manager／検索基盤）

ベクトル検索と SQL の強みを両取りする。この会話で固めた **「ベクトルで広げ、SQL で正本に収束させる」** 順序をここに反映する。

| ID | 中分類 | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|---|
| E-1 | ベクトル検索 | 関連しそうな正本候補を広く想起（pgvector） | 📋 計画中 | 原則4 |
| E-2 | SQL 絞り込み | 候補を `status='active'` かつ有効期限内に機械的に収束 | 🔨 実装予定 | 原則1・2 |
| E-3 | 全文検索 | キーワードによる正本の全文検索 | 📋 計画中 | 原則4 |
| E-4 | 文脈前面化 | セッションで参照する正本を選び前面に出す | 🔨 実装予定 | 原則1 |
| E-5 | 参照範囲制御 | AI にどの正本を渡すかを制御（曖昧化の防止） | 🔨 実装予定 | 原則2 |

> **設計メモ**：E-1（ベクトル検索）は pgvector 移行が前提のため Phase 1.5。Phase 1.0 は E-2 の SQL 絞り込みだけで「正本だけを参照する」を成立させる。ベクトルを先に、SQL を後に置く順序が肝で、逆にすると古い情報を拾いハルシネーションの温床になる。

### F. 監査・履歴・可視化（Audit Log／履歴ビュー）

全操作を記録し、かつ **「AI がどう情報を認識しているか」を人間が確認できる**ようにする。使用者が正本を閲覧・修正できることは、E.G.O の設計上の必須要件。

| ID | 中分類 | 小分類（機能） | 状態 | 対応原則 |
|---|---|---|---|---|
| F-1 | 監査ログ | 全操作（登録・承認・遷移）の記録 | 🔨 実装予定 | 原則1 |
| F-2 | 履歴閲覧 | 要約ビュー（正本テーブルのみ／1 トピック 1 行） | 📋 計画中 | 原則1・4 |
| F-3 | 履歴閲覧 | トピック単位の履歴展開（タップで当該トピックの改訂履歴を時系列表示） | 📋 計画中 | 原則1・4 |
| F-4 | 正本の人間管理 | 使用者による正本の閲覧・修正 | 📋 計画中 | 原則1・2 |
| F-5 | 参照フットプリント | AI 応答に参照した正本の ID・タグ・状態を添付 | 📋 計画中 | 原則2・4 |
| F-6 | 認識可視化 | フットプリントの ID から正本行へ直接遷移 | 📋 計画中 | 原則2・4 |
| F-7 | 解釈過程の可視化 | AI の解釈プロセスそのものの可視化 | 📋 計画中 | 原則2 |

> **設計メモ**：F-2〜F-3 の三階層（要約ビュー→トピック履歴→監査ログ）は、「情報量が多すぎて人間が管理できない」という課題への回答。日常的に目に触れるのは要約ビューだけにし、深掘りは段階的に開く。F-5〜F-6 の参照フットプリントが「自分が自分を知る」（原則4）を UI 上で成立させる中核。

## 4. 分類の設計判断（付記）

- **なぜこの 6 層か**：E.G.O 本来の 4 レイヤー（入力／Core／データ／LLM）に対し、Core を機能単位（思考構造化・正本管理・承認・検索・監査）へ展開した。これは、機能を「入口・中核・記憶・拡張・運用」で層状に束ねる Hermes Agent の整理方針を参考にしつつ、E.G.O の核心（C. 正本管理）が埋もれないよう独立の大分類として立てたもの。
- **Hermes から借りたもの／借りなかったもの**：借りたのは「機能を層で束ねる並べ方」。借りなかったのは Hermes の機能セットそのもの（自己改善ループ、cron、6 種バックエンド等）。E.G.O は多機能さではなく正本管理の一点で価値を出すため、機能数を意図的に絞っている（原則3）。
