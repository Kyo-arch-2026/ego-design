## 0. 読み方

- **層（大分類）**：E.G.O の 4 レイヤー構成（入力層／Core／データ層／LLM 層）を基準に、機能を束ねている。分類の考え方は、機能を「入口・中核・記憶・拡張・運用」で層状に整理する Hermes Agent の構造を参考にした。ただし後述のとおり、E.G.O は Hermes 本体に依存せず、疎結合な境界を保つ設計とする。
- **状態**の凡例：
  - 📐 **設計済**：設計ドキュメントで方針が確定している。コードは未着手。
  - 🔨 **実装予定（Phase 1.0）**：Phase 1.0 でコードを書く対象。
  - 📋 **計画中（Phase 1.5以降）**：将来フェーズに予定し、現時点で未着手。
- **現状の前提**：本書作成時点で、E.G.O は **設計フェーズ**にあり、コードや動作するものはまだ存在しない。GitHub 上の設計ドキュメントが唯一の成果物である。したがって本書の状態表記は「どこまで設計が固まり、どこから実装・構想か」を示すものであり、動作状況を示すものではない。
- **対応原則**：各機能が E.G.O の 4 原則のどれを根拠に存在するかを示す。
  - 原則1＝意志の文脈を劣化させない／原則2＝LLM（大規模言語モデル）は補助役に徹する／原則3＝常駐コンポーネントを増やさない／原則4＝使うほど自分への解像度が上がる

## 0.1 アーキテクチャ方針：疎結合と抽象化（本書全体の前提）

E.G.O は「機能性こそが能動性の前提」という思想を持ち、手間（保守・バージョン管理・障害切り分け）を減らすことを重視する。この思想は、アーキテクチャとして次の方針に落ちる。

- **外部技術に依存しない疎結合構造**：入力元・LLM プロバイダー・ストレージという「外部と接する 3 つの層」は、いずれも抽象インターフェースで隔離し、具体的な技術（Hermes／Claude／SQLite など）はその背後のアダプタに閉じ込める。E.G.O Core はアダプタの背後にある実体を知らない。
- **なぜこうするか**：(1) 特定技術の更新に E.G.O 全体が振り回されるのを防ぐ（バージョン管理の手間削減）。(2) 障害が起きた層をアダプタ単位で切り分けられる（障害点の切り分け容易化）。(3) 新しい入力元・プロバイダー・ストレージをアダプタ追加だけで足せる（後からの機能追加容易化）。以上は原則3 と E.G.O の手間削減思想の直接の帰結である。
- **Hermes の位置づけ**：入力層は Hermes 本体に依存しない。Hermes は「入力アダプタの一実装（選択肢）」として境界の外に置く。Hermes の有用な機能が必要な場合は、その機能をアダプタ経由で限定的に借りる。Hermes 本体を E.G.O に組み込んで従属させることはしない。

この方針を、各層の機能（A：入力、C/データ：ストレージ、LLM）に一貫して適用する。

## 2. 機能分解（詳細）

### A. 入力・インターフェース層

ユーザーとの接点。**特定の入力技術に依存せず、抽象インターフェース（Input Port）を介して Core へ入力を渡す。** CLI・Discord・Hermes などは、この Port に適合する「入力アダプタ」として実装する。

| A-2 | 入力アダプタ | Discord／Telegram 入力アダプタ | 📋 計画中 | 原則3 |
| A-3 | 入力アダプタ | 音声メモ文字起こし入力アダプタ | 📋 計画中 | 原則3 |
| A-4 | 入力アダプタ | Hermes 入力アダプタ（Hermes の機能を借りる場合のみ） | 📋 計画中 | 原則3 |

| A-6 | セッション連続性 | 入力元をまたいだ会話の継続 | 📋 計画中 | 原則1 |

> **設計メモ**：Phase 1.0 は CLI アダプタのみを実装する。Hermes は必須依存ではなく、A-4 の「Hermes アダプタ」という一選択肢に格下げした。これにより Hermes のバージョン更新が E.G.O Core に波及しない（原則3・手間削減）。入力元が増えても、Core は Input Port しか見ないため無改修で済む。

| B-2 | 構造化 | 課題の抽出 | 🔨 実装予定 | 原則2 |
| B-3 | 構造化 | 選択肢の列挙 | 🔨 実装予定 | 原則2 |
| B-4 | 構造化 | 次のアクションの提示 | 🔨 実装予定 | 原則2 |
| B-5 | 原文保持 | 構造化前の元テキストを原文として保持 | 🔨 実装予定 | 原則1 |

### C. 正本管理・状態遷移（Source of Truth）★ E.G.O の核心

情報を上書きせず、状態を遷移させることで「今有効な正本」だけを参照可能にする。正本テーブルと履歴テーブルを分離する。**永続化は必ず Store Port（G-3）経由で行い、Source of Truth は SQLite 等の具体的なストレージを直接触らない。**

#### C-1. データ構造

|---|---|---|---|

| C-1-2 | 履歴テーブル（`fact_revisions`）：candidate／superseded／invalid／archived を保持する追記専用ログ | 🔨 実装予定 | 原則1 |
| C-1-3 | 事実カード型の主キー設計（1 判断＝1 カードを基本単位とする） | 🔨 実装予定 | 原則4 |
| C-1-4 | トピック束ねタグ（`topic_tag`）による後付けグルーピング用カラム | 📋 計画中 | 原則4 |

|---|---|---|---|

## 4. 分類の設計判断（付記）

- **なぜ G 層（抽象化・アダプタ基盤）を独立させたか**：入力・LLM・ストレージの抽象化は、E.G.O の疎結合・手間削減・障害切り分けという非機能目標を構造的に支える中核である。これを各層に埋もれさせず独立の大分類として立てることで、「E.G.O は外部技術に依存しない」という設計思想を一貫して主張できる。
- **Hermes から借りたもの／借りなかったもの**：借りたのは「機能を層で束ねる並べ方」という整理の発想のみ。Hermes 本体には依存せず、Hermes は入力アダプタの一選択肢（A-4）として境界の外に置いた。これは Hermes のバージョン更新に E.G.O が振り回されるのを防ぎ、疎結合を保つためである（原則3・手間削減）。
