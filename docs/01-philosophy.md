# 01. Philosophy — 4つの設計原則

E.G.Oの設計を貫く4つの原則です。すべての技術選定と機能はこの原則から導かれます。

## 原則1：意志の文脈を劣化させない

人が考えたことを、時間と共に劣化させない。過去の判断と現在の判断を同じ重みで扱える状態を保つ。

銀行の監視業務では、ベテランの経験則がマニュアルに記載されず、その人にしか分からない判断として消えていく場面を見てきた。記録されない判断は、確認も継承もできない。

設計への影響：
- データは上書きせず履歴を残す（candidate → active → superseded）
- 全操作を監査ログとして記録する

## 原則2：LLMは補助役に徹する

LLMを「判断する主体」にしない。最終判断は必ず人間が行う。

AIの解釈と自分の考えが混ざった時、「これは本当に自分の考えか」が分からなくなる。便利に使いながら、その境界を保つ必要がある。

設計への影響：
- AIが構造化した内容は、人間が承認して初めて正本になる
- LLMプロバイダーは抽象化し、特定企業に依存しない

## 原則3：常駐コンポーネントを増やさない

機能を追加する時、新しいサービスやDBを増やさない。プラグインとして拡張する。

設計への影響：
- 拡張型モジュラーモノリスを採用
- DBはPostgreSQL + pgvectorに集約する

## 原則4：使うほど自分への解像度が上がる

蓄積された判断の歴史から、自分の価値観が可視化される。AIが自分を知るのではなく、自分が自分を知るための土台。

設計への影響：
- 採用した結論を蓄積する
- 判断基準を明文化する
