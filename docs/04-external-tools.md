# 04. External Tools — 外部ツール選定の設計

E.G.Oは「車輪の再発明をしない」を原則とする。外部ツールを選ぶ時も「なぜそれか」「なぜ他はダメか」を明示する。

ただし E.G.O は、外部ツールを Core に直接組み込まず、**抽象ポートの背後にアダプタとして配置する**。これにより、特定ツールへの依存を避け、そのツールのバージョン更新が Core に波及しないようにする（原則3・疎結合）。以下の選定も、この前提のもとで読むこと。

## 1. 入力基盤：Hermes Agent（入力アダプタの一選択肢）

**Hermes 本体を E.G.O に組み込むことはしない。** Hermes は Input Port に適合する「入力アダプタの一選択肢」として、境界の外に置く。Hermes の有用な機能（セッションをまたぐ記憶など）が必要になった場合に、その機能をアダプタ経由で限定的に借りる。

Hermes Agentは、Nous Researchが開発したオープンソースのAIエージェント基盤。セッションをまたいだ記憶と、プロバイダー非依存を設計の核に持つ。

なぜ Hermes を「借りる候補」として評価するか：

- セッションをまたぐ記憶 → 原則1（意志の文脈を劣化させない）と一致
- プロバイダー非依存 → 原則2（LLMは補助役）と一致

なぜ本体依存にしないか：

- Hermes は大規模（多数のツール・複数の実行バックエンドを持つ）で、本体に組み込むとそのバージョン更新に E.G.O が振り回される（保守の手間増）
- 障害が起きたとき、Hermes 由来か E.G.O 由来かの切り分けが難しくなる（疎結合の目標に反する）
- E.G.O の核心は正本管理であり、入力層に巨大な依存を抱えるのは設計上重い（原則3）

したがって、Phase 1.0 の入力は自前の CLI アダプタで実装し、Hermes アダプタは必要になった時点（Phase 1.5 以降）で追加する。

なぜ他のツールではないか（入力アダプタの候補として）：

| ツール | 不採用理由 |
| --- | --- |
| OpenClaw | Hermesへの公式移行パスがあり、Hermesが後継 |
| 自前フルスクラッチの入力基盤 | 入力層より正本管理に集中すべき（原則3）。ただし Phase 1.0 の CLI は薄いアダプタとして自前実装する |
| LangChain | フレームワーク依存が強くロックインのリスク。疎結合の方針に反する |

## 2. データベース：PostgreSQL + pgvector（Store Port のアダプタ）

**Store Port の背後に置くアダプタとして採用する。** Core は Store Port だけを知り、SQLite か PostgreSQL かを知らない。

採用理由：

- 正本管理に必要な書き込みの信頼性（ACID）が高い
- pgvectorでベクトル検索もできる
- 1つのDBで全文検索・ベクトル検索・正本管理が完結

なぜ他ではないか：

| 選択肢 | 不採用理由 |
| --- | --- |
| SQLite | 本番実績と信頼性で劣る。ただし Phase 1.0 では検証目的に十分なため、Store Port の Phase 1.0 アダプタとして採用する |
| MongoDB | 正本管理に必要なトランザクションの堅牢性が弱い |
| 専用ベクトルDB（DynamoDB等） | キー検索が前提で複雑な状態管理に不向き。DB種別を増やしたくない（原則3） |

フェーズ方針：Phase 1.0 は SQLite アダプタで実装する。Phase 1.5 で PostgreSQL+pgvector アダプタを追加し、Store Port 経由で切り替える。Core は改修しない。

## 3. LLM：抽象化レイヤー（LLM Port）で切り替え可能に

**LLM Port の背後にアダプタとして配置し、特定プロバイダーに依存しない。** Core は LLM Port だけを知る。

| 用途 | プロバイダー | フェーズ |
| --- | --- | --- |
| 深い思考 | Claude | 1.0 |
| 前処理・分類 | cheap LLM | 1.5 |
| ローカル実行 | Ollama | 1.5 |

Phase 1.0 は Claude アダプタのみ。cheap LLM・Ollama アダプタは Phase 1.5 で追加する。

## 4. なぜ個人開発でAWSを使わないか

E.G.Oは個人の判断データを扱うため、データを手元に置く設計を選んでいる。これは「常駐コンポーネントを増やさない」（原則3）という設計原則とも一致する。

ただし、将来複数デバイスでの利用が必要になった場合は、RDS（PostgreSQL）+ EC2/Lambdaへの移行パスを想定している。この移行も Store Port へのアダプタ追加で吸収し、Core を改修しない。今は不要な常駐コストを避けるという、コスト合理性に基づいた判断である。

## まとめ：選定を貫く一貫した考え方

外部ツールは「Core に組み込む」のではなく「ポートの背後にアダプタとして置く」。これにより、どのツールも差し替え可能な選択肢に留まり、E.G.O が特定技術に縛られない。ツール選定の良し悪しが Core の設計を汚さないことが、この構造の狙いである（疎結合・原則3）。
