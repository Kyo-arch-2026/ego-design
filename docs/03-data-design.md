# 03. Data Design — データ設計の核心

E.G.O のデータ設計は「情報に時間軸を持たせ、常に今有効な正本だけを参照する」ことを核とする。既存の LLM が過去の情報をフラットに扱い、古くなった判断もそのまま参照してしまう問題を、状態遷移と正本の分離によって解決する。

## なぜデータ設計が核心なのか

E.G.O の目的は「これは本当に自分の考えか」を確認できることである。そのためには、AI がハルシネーションを起こさず「今有効な正本」だけを参照し、かつ人間がその正本を閲覧・修正できなければならない。この二つを両立させるのがデータ設計の役割である。

ここで相反する二つの要求が生じる。

- **AI 側の要求**：参照する正本は「今有効な 1 件」に絞られていてほしい。曖昧だとハルシネーションが起きる。
- **人間側の要求**：意志を劣化させないため、候補や過去の正本を含む全履歴を閲覧・修正できなければならない。

「AI が見る正本の量」と「人間が管理する情報の総量」は本質的に別物である。これを一つのビューで解こうとすると破綻する。したがって、両者を分離するのが設計の骨子となる。

## 正本テーブルと履歴テーブルの分離

役割で物理的にテーブルを分ける。

- **正本テーブル（`canonical_facts`）**：`status='active'` のものだけを保持する、常に小さいテーブル。AI とベクトル検索が参照するのはここだけ。1 カード＝1 行を保つ。
- **履歴テーブル（`fact_revisions`）**：candidate / superseded / invalid / archived を保持する、追記専用（append-only）のログ。人間が「なぜ変わったか」を追うのはここ。

状態遷移は「行の上書き」ではなく「正本テーブルの 1 行を差し替え、旧行を履歴テーブルへ退避する」操作とする。これにより、AI は常に軽く、履歴は完全に残る、が両立する。

## 事実カード型で始める

正本の基本単位は「事実カード型」（1 判断＝1 カード）とする。

普段使いの入力は「今こう思った」「この結論に至った」という単発の粒で発生する。これはカード型と自然に一致し、入力時の摩擦が少ない。トピック単位でまとめたい場合は、カードに `topic_tag` を付け、ビューで束ねることで後付けできる。テーブル構造を変えずにトピック型の見え方が得られる。

主キーはカード側に置く。粗い単位（トピック）を主キーにすると後で細かくできないが、細かい単位（カード）を主キーにすれば後からいくらでも束ねられる。型（カード型／トピック型）は使用感とレイテンシーを見て確定する方針であり、現時点では決め打ちしない。

## 状態遷移

正本は直接上書きせず、状態を遷移させる。

```
candidate（候補） → active（有効） → superseded（置換済）
                                   ↓
                          invalid / archived
```

- candidate：AI が構造化した、まだ承認されていない情報。正本ではない。
- active：人間の承認を経て正本になった状態。
- superseded：新しい正本に置き換えられた過去の正本。履歴として保持される。
- invalid / archived：無効化・保管された状態。

これは原則1（意志の文脈を劣化させない）の具体的な実装である。

## ベクトル検索と SQL の役割分担

検索は「ベクトルで広げ、SQL で正本に収束させる」順序で行う。

1. **ベクトル検索**：関連しそうな正本候補を広く想起する。あいまい検索・想起に強い。ただしこの結果はまだ信じない。
2. **SQL 絞り込み**：想起した候補に対し `status='active'` かつ有効期限内という条件で機械的に絞り込む。ここで初めて「今信じてよい正本」が確定する。

順序が重要である。ベクトルで広げ、SQL で収束させる。逆にすると古い情報や候補を拾い、ハルシネーションの温床になる。「DB を唯一の正本と認識させる」という狙いは、この SQL 収束の段で担保される。

なお Phase 1.0 はストレージを SQLite で代替するためベクトル検索を持たない。SQL 絞り込みだけで「正本だけを参照する」を成立させる。ベクトル検索は pgvector へ移行する Phase 1.5 で導入する。

## 人間が溺れないための三階層ビュー

AI が正本を唯一の正本として認識する以上、人間がその正本を閲覧・修正できることは必須要件である。しかし全履歴をそのまま見せると、情報量が使用者の管理できる範囲を超える。そこで粒度を三階層に分け、段階的に開く。

- **要約ビュー（デフォルト）**：正本テーブルのみ。1 トピック 1 行。日常的に目に触れるのはここだけ。
- **トピック単位の履歴**：あるトピックを開いたときだけ、その改訂履歴を時系列で表示する。
- **監査ログ**：全操作の記録。普段は畳んでおき、必要時のみ開く。

日常的に見えるのは要約ビューだけにし、深掘りは段階的に開く。これが「情報量が多すぎて人間が管理できない」という課題への回答である。

## AI の認識を人間が確認する：参照フットプリント

AI が応答を返すとき、参照した正本の ID・タグ・状態を応答に添付する（参照フットプリント）。使用者はその ID から正本テーブルの当該行へ直接飛び、閲覧・修正できる。

これにより「AI が何を正本として見て答えたか」が毎回追跡可能になる。「AI が自分を知る」のではなく「自分が自分を知る」（原則4）が、確認可能な形で成立する。

## 4 原則との対応

- 履歴を消さず追記（`fact_revisions`）→ 原則1
- AI は正本を参照するだけ、修正権限は人間 → 原則2
- テーブルは単一 DB に集約し、DB 種別を増やさない → 原則3
- 参照フットプリントで判断の来歴が見える → 原則4
